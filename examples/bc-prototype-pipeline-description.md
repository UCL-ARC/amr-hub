# BC Prototype: Strategy and Pipeline

A behavioural cloning prototype for HCW movement on RTLS data. The goal is to test whether a learned policy can reproduce real HCW behaviour well enough to justify moving to inverse RL — and to identify the minimum state representation that supports this.

## Problem

We have a short real trajectory from one HCW (event log, ~20 rows). This is too little to train BC directly. We need to (a) inflate the data without inventing structure, (b) decide what features the policy should see, and (c) measure whether the trained policy actually captures real behaviour.

## Pipeline

```text
Real CSV  →  Resample to 15-min grid  →  Fit Markov bootstrap  →  Sample synthetic shifts
                                                                          ↓
                                           Train BC per (state, action) variant
                                                                          ↓
                                           Evaluate on real held-out (accuracy + rollout TV)
                                                                          ↓
                                              Occam recommendation → IRL
```

## 1. Resampling (Option A)

Event-level CSV is sparse; BC needs a regular time series. We resample each shift to a 15-minute grid:

- **Zone:** forward-filled. The HCW stays where they were until the next event moves them.
- **Event type:** sparse marker. Sustained events (`workstation`, `attend_patient`) forward-fill until the next event; transient events (`door_access`) fire for one grid point and decay to `none`; terminal events (`occupy_content`) end the shift.
- **Shift splitting:** on gaps > 8 hours OR on terminal events.

Fields used: `timestamp`, `location`, `event_type`. Fields deliberately ignored for now: `patient_id`, `door_id`, `content_type`.

## 2. Synthetic data via Markov bootstrap

Two empirical distributions are fit on the real seed:

- **`P(next_zone | curr_zone, time_bucket)`** — zone transitions conditioned on time of day. Laplace-smoothed.
- **`P(event_type | zone)`** — per-zone event marginal. Terminal events excluded.

Synthetic shifts are generated by sampling auto-regressively: at each grid step, sample the event from `P(event | zone)`, sample the next zone from `P(next_zone | curr_zone, time_bucket)`, advance the clock. 200 synthetic shifts × 80 steps each.

**Why these variables and not patient_id:** zone-to-zone transitions reflect stable hospital structure (corridors, room layout, workflow) and are estimable from a short seed. Patient identity is not — it depends on the day's care plan, not on the previous patient, and isn't well-described by a transition model. Patient identity re-enters at IRL as a reward feature.

**Conditional independence caveat:** events and next-zone are sampled independently given the current state. In real data they're correlated (finishing documentation predicts moving differently than finishing a patient visit). Capturable later as `P(next_zone | curr_zone, time_bucket, curr_event)` once more data is available.

## 3. BC training: state/action sweep

Because we don't know upfront which features matter, we train one model per combination:

**State variants** (Occam-ordered, smallest first):

- `S1`: `[curr_zone]`
- `S2`: `[curr_zone, time_bucket]`
- `S3`: `[curr_zone, time_bucket, prev_zone]`
- `S4`: `S3 + [curr_event]`
- `S5`: `S4 + [dwell_bin]`

**Action variants:**

- `A1`: `next_zone`
- `A2`: `next_event`
- `A3`: joint `(next_zone, next_event)`

Each variant trains a small Flax MLP with per-feature embeddings on synthetic + real-train data. ~30 epochs, Adam, cross-entropy.

## 4. Evaluation

Two metrics per variant, both computed on real held-out only:

1. **Step-by-step accuracy.** Given the true state at each held-out step, did the model predict the true action? Top-1 and top-3.
2. **Distributional match.** Auto-regressively roll out the model from the held-out shift's first state. Compare zone and event histograms (rollout vs real held-out) via total variation distance.

These measure different things and can disagree. Step-by-step is strict; distributional is what matters for AMR, where transmission depends on aggregate co-location patterns rather than per-step prediction accuracy.

## 5. Occam recommendation

The "best" variant is the smallest one that captures real structure:

- **Distributional filter:** smallest state where `TV(zones) ≤ 0.20` AND `TV(events) ≤ 0.20`.
- **Elbow filter:** smallest state where the next-larger variant adds < 2% top-1 gain.

When the two filters agree, the recommendation is unambiguous. When they disagree, the table is presented and the choice is a modelling judgment.

## Discipline

- **Synthetic data is for training only.** Held-out evaluation uses only real data, so the pipeline cannot win by matching its own generator's quirks.
- **Synthetic data is a scaffold, not a destination.** As real data grows, the synthetic share of training data shrinks, and eventually the bootstrap step disappears.
- **Inspection artefacts.** `--save-data` writes resampled real, synthetic, transition tables, and rollouts to CSV for sanity-checking with collaborators.

## Outputs

Running the sweep produces:

- A comparison table across all `(state, action)` variants with top-1, top-3, TV-zones, TV-events, and pass/fail.
- An Occam recommendation for the state representation to carry into IRL.
- Saved CSVs (with `--save-data`) for the resampled real shifts, synthetic shifts, Markov transition tables, per-zone event distributions, and one rollout per variant alongside the real held-out shift for direct comparison.

## What this is and isn't

This is the **Step 1 sanity check** of the AMR-HUB roadmap. It is not a deployable movement model — BC is brittle off-distribution and can't respond to disease state. Its purpose is to validate that the state representation and the data preprocessing are good enough to support inverse RL in Step 2. If BC can't match real held-out behaviour distributionally, IRL on the same representation will silently produce a plausible-looking but meaningless reward function.
